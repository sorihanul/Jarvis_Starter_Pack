# Codex Agent Utilization v0.1

## 목적
- `Jarvis Starter`를 Codex에서 운용할 때, **에이전트 기능을 어디까지 어떻게 쓰는지** 최소 규칙으로 고정한다.

## 전제
- `Jarvis Starter` 본체는 모델 독립이다.
- Codex는 실행 환경 중 하나다.
- 따라서 Codex 기능은 `정체성`이 아니라 `호환 실행층`으로만 취급한다.
- 이 문서는 운영 가이드이며, 실제 Codex 프로젝트 지침은 `AGENTS.md` 계열, 로컬 역할 설정은 `.codex/config.toml` 계열에서 관리한다.

## 1. 언제 서브에이전트를 열 것인가

아래 중 하나일 때만 권장:
1. 읽기 작업과 구현 작업을 분리하면 품질이 올라갈 때
2. 검토와 구현을 같은 세션이 동시에 들고 있으면 드리프트가 커질 때
3. 문서 조사, 구현, 검증을 병렬화하면 시간이 줄어들 때
4. 메인 세션이 정체성/통합 판단에 집중해야 할 때

아래는 기본적으로 단일 세션 유지:
1. 짧은 질의응답
2. 단일 파일 소규모 수정
3. 구조 논의만 있는 경우
4. 브리핑이나 개요 작성

## 2. 기본 역할 분리

Codex에서 가장 안전한 기본 분리는 아래 4가지다.

1. `research`
- 문서/코드/맥락 탐색
- 출력: 사실, 경로, 근거

2. `implementation`
- 실제 수정안 작성
- 출력: 변경된 파일과 핵심 구현 포인트

3. `validation`
- 테스트, 검증, 리스크 확인
- 출력: 통과/실패와 남은 위험

4. `synthesis`
- 결과 통합, 최종 보고 정리
- 출력: 사용자용 최종 응답 초안

## 3. 운영 원칙

1. 메인 세션은 항상 통합자다.
2. 서브에이전트는 좁고 명확한 일만 맡는다.
3. 읽기 중심 역할을 먼저 분리한다.
4. 구현 역할은 가능한 한 파일 경계를 분명히 한다.
5. 검증 역할은 구현을 다시 하지 않고, 점검에 집중한다.
6. 서브에이전트 결과를 그대로 내보내지 않고 메인 세션이 통합한다.
7. 프로젝트형 큰 작업은 메인 세션에서 바로 밀기보다 별도 오케스트레이션 세션에서 roster를 먼저 정한다.

## 4. 스타터 팩과의 정합 규칙

1. 정체성 레이어(`START_HERE.md`, `MAP.md`, `POLICY.md`, `00_Core`)는 그대로 유지한다.
2. 작업 산출은 여전히 `TASKS/`, `CAPSULES/`, `LOGS/`에 남긴다.
3. 코어 파일 변경은 별도 승인 원칙을 유지한다.
4. Codex 에이전트 기능은 `프로젝트 분업`일 뿐, 자비스 본체를 대체하지 않는다.
5. `AGENTS.md`는 작업 규칙 표면이고, `.codex/config.toml`은 로컬 실행 표면이다.

## 5. 추천 흐름

### 가벼운 작업
- 메인 세션 단독

### 표준 작업
- 메인 세션
- 필요 시 `research` 또는 `validation` 1개만 추가

### 큰 작업
- 메인 세션
- 필요 시 별도 오케스트레이션 세션에서 역할 배치 결정
- `research`
- `implementation`
- `validation`
- 마지막에 메인 세션이 `synthesis`

## 6. 권장 파일 표면

Codex에서 자비스 스타터를 쓸 때 표면은 아래처럼 나누는 편이 가장 안정적이다.

1. `AGENTS.md`
- 프로젝트 공통 작업 규칙
- 읽기 순서, 산출 위치, 금지사항

2. `AGENTS.override.md`
- 특정 하위 폴더 또는 프로젝트 전용 규칙
- 프로젝트 오케스트레이션 세션용 추가 규칙

3. `.codex/config.toml`
- 로컬 Codex 실행 설정
- 멀티에이전트 깊이, 스레드 수, 역할 description

즉 `무엇을 하게 할지`는 `.md`, `Codex가 어떻게 불릴지`는 `.toml`로 나눈다.

### 표준 역할 파일 형식

- 표준 역할 설명 문서는 있을 수 있다.
- 하지만 실제 Codex 역할 파일은 `MD`가 아니라 `TOML`로 두는 편이 맞다.
- 공개 기준에서 가장 안전한 표면은 `.codex/agents/*.toml` 같은 standalone custom agent 파일이다.
- 따라서 `research`, `implementation`, `validation`, `synthesis`, `security_gate`, `writing` 같은 표준 역할은 설명용 문서만 두기보다 실제 `*.toml` 파일 예시를 같이 제공하는 편이 좋다.
- `agents.<name>.config_file` 같은 조립형 패턴은 로컬 관행 예시로만 보고, 현재 Codex 공개 문서와 다를 수 있음을 전제로 사용한다.

## 7. 금지

1. 작은 작업도 습관적으로 서브에이전트부터 여는 것
2. 서브에이전트에게 정체성 판단까지 넘기는 것
3. 코어 레이어를 서브에이전트가 무심코 수정하게 두는 것
4. 병렬화 자체를 성숙도로 착각하는 것
5. 자비스 문서와 Codex 로컬 설정을 한 파일에 섞어 관리하는 것
6. 실제 Codex 역할 파일을 `MD`로 만들어 놓고 설정 파일처럼 오해하게 두는 것

## 결론
- Codex의 에이전트 기능은 강력하다.
- 하지만 `Jarvis Starter`에서는 어디까지나 **선택적 분업 레이어**로만 붙인다.
- 메인 세션은 항상 자비스 본체의 판단과 통합을 유지한다.
