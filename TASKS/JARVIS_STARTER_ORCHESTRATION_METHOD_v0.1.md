# Jarvis Starter Orchestration Method v0.1

## 목적
- 자비스 스타터의 여러 문서형 자산을 어떻게 함께 사용할지 설명하는 경량 방법론을 정의한다.
- 공개용 스타터가 제품처럼 굳어지지 않도록, 강제 시스템이 아니라 선택 가능한 운영 패턴으로 제시한다.
- 초보 사용자는 기본 흐름만 따라도 시작할 수 있고, 숙련 사용자는 세미 오토, 보안 게이트, 학습 루프를 선택적으로 얹을 수 있게 한다.

## 이것이 무엇인가
- 스타터 내부 자산들을 연결하는 상위 운영 가이드
- 문서형 스타터를 실제 작업 흐름으로 엮는 방법론
- 프로젝트, 세션, 학습, 보안, 선택 모듈을 배치하는 기준

## 이것이 무엇이 아닌가
- 강제 런타임 스펙이 아니다
- 특정 IDE나 실행기를 전제하지 않는다
- 사용자의 개인 시스템을 대체하지 않는다
- 반드시 따라야 하는 정답 운영체계가 아니다

## 한 줄 원칙
자비스 스타터는 사용자를 대신하는 완성형 시스템이 아니라, 사용자가 자기 에이전트 시스템을 조립할 때 쓰는 안전한 작업 레일이다.

## 세션 분리 원칙
- 메인 세션과 프로젝트 오케스트레이션 세션은 분리하는 것을 권장한다.
- 메인 세션은 스타터 자체를 설계, 유지, 정리하는 비서/브레인 역할을 맡는다.
- 프로젝트 오케스트레이션은 별도 세션을 열어 진행하는 것을 기본 권장 방식으로 둔다.
- 이유:
  - 스타터 설계와 프로젝트 진행이 섞이지 않게 하기 위해
  - 프로젝트별 에이전트 소환과 흐름 설계를 독립적으로 유지하기 위해
  - 작업 기록과 학습 기록의 오염을 줄이기 위해

### 메인 세션의 역할
- 스타터 코어와 선택 모듈을 읽고 정리한다.
- 에이전트/스킬/템플릿 자산을 설계하고 유지보수한다.
- 공개판 승격 여부와 문서 방향을 판단한다.
- 특정 프로젝트의 상세 집행은 직접 맡지 않는다.

### 프로젝트 오케스트레이션 세션의 역할
- 특정 프로젝트를 시작할 때 별도로 연다.
- 프로젝트 목적, 모드, 에이전트 roster, 단계 흐름, 산출물 구조를 정한다.
- 프로젝트 문서와 상태를 기준으로 실제 진행을 지휘한다.
- 필요 시 메인 세션의 자산을 참조하지만, 메인 세션 자체를 작업장으로 쓰지는 않는다.

## 빠른 진입
### 사람용 빠른 시작
1. `README.md`를 읽고 이 스타터의 성격을 이해한다.
2. `START_HERE.md`를 읽고 기본 부팅 순서를 확인한다.
3. 지금 필요한 작업이 단발인지 프로젝트인지 판단한다.
4. 단발이면 바로 메인 세션에서 시작하고, 프로젝트면 별도 오케스트레이션 세션을 연다.
5. 프로젝트 오케스트레이션 세션에서는 `TASKS/PROJECTS/` 또는 프로젝트 전용 작업장을 기준으로 진행한다.
6. 필요할 때만 선택 모듈을 추가로 읽는다.

### 에이전트 진입 경로
에이전트는 아래 순서로 읽고 진입한다.

1. `START_HERE.md`
- 현재 워크스페이스가 자비스 스타터임을 인식한다.
- 작업은 `TASKS`, `CAPSULES`, `LOGS`에만 남긴다는 경계를 먼저 잡는다.

2. `MAP.md`
- 폴더 구조와 계층 역할을 파악한다.
- 어떤 문서가 코어이고 어떤 폴더가 작업 레이어인지 구분한다.

3. `POLICY.md`
- 출력 형식, 승인 원칙, 실행/대화 분리 규칙을 확정한다.
- 코어 보호와 작업 레이어 분리를 최우선 규칙으로 건다.

4. `AGENT_INDEX.md`와 `SKILL_INDEX.md`
- 현재 사용할 수 있는 역할과 스킬을 파악한다.
- 요청에 맞는 에이전트/스킬 후보를 좁힌다.

5. 현재 요청 분류
- 단발 세션이면 바로 현재 요청을 처리한다.
- 프로젝트형이면 별도 오케스트레이션 세션을 여는 것을 우선 권장한다.
- 프로젝트 오케스트레이션 세션에서는 `TASKS/PROJECTS/<project_id>/` 작업장을 사용한다.
- 추가 흐름 제어가 필요하면 선택 모듈을 읽는다.

### 에이전트 선택 로드 규칙
- 세션 흐름 정리가 필요하면 `01_Modules/Madang_Pan_Lite_Starter/README.md`
- 반자동 운영이 필요하면 `TASKS/SEMI_AUTO_PRESET_v0.1.md`
- 프로젝트 작업장이 필요하면 `01_Modules/Project_Workspace_Lite_Starter/README.md`
- 세션 학습이 필요하면 `01_Modules/Learning_Loop_Lite_Starter/README.md`
- 보안 게이트가 필요하면 `TASKS/SECURITY_GATE_AGENT_SPEC_v0.1.md`

### 에이전트 진입 원칙
1. 코어를 읽고 작업 레이어로 내려온다.
2. 요청 분류 전에는 선택 모듈을 과하게 읽지 않는다.
3. 필요한 모듈만 최소 로드한다.
4. 코어 문서에 작업 내용을 기록하지 않는다.
5. 작업이 끝나면 캡슐 또는 승격 후보로만 압축해 남긴다.

## 구성 요소
### 1. Identity Layer
- 목적: 에이전트의 기본 판단 규칙과 경계 설정
- 주요 문서:
  - `START_HERE.md`
  - `MAP.md`
  - `POLICY.md`
  - `00_Core/`

### 2. Task Layer
- 목적: 실제 작업, 임시 산출물, 프로젝트 문서, 캡슐, 로그 기록
- 주요 경로:
  - `TASKS/`
  - `CAPSULES/`
  - `LOGS/`

### 3. Method Layer
- 목적: 작업을 어떻게 진행할지에 대한 경량 운영 패턴 제공
- 주요 자산:
  - `01_Modules/Madang_Pan_Lite_Starter/`
  - `01_Modules/Project_Workspace_Lite_Starter/`
  - `01_Modules/Learning_Loop_Lite_Starter/`
  - `TASKS/SEMI_AUTO_PRESET_v0.1.md`
  - `TASKS/SECURITY_GATE_AGENT_SPEC_v0.1.md`

### 4. Asset Layer
- 목적: 반복 가능한 절차와 역할을 자산으로 승격할 후보를 모으는 층
- 주요 경로:
  - `AGENTS/`
  - `SKILLS/`
  - `TASKS/PROMOTION_CANDIDATE_TEMPLATE.md`

## 기본 오케스트레이션 흐름
1. Identity Layer를 읽고 작업 경계를 확정한다.
2. 요청을 작업군으로 분류한다.
3. 필요 시 IVK로 관련 자료와 과거 작업을 회수한다.
4. 작은 작업이면 세션 단위로 바로 처리한다.
5. 프로젝트형 작업이면 별도 오케스트레이션 세션을 열고 `TASKS/PROJECTS/<project_id>/` 작업장으로 정렬한다.
6. 위험한 입력과 출력은 보안 게이트로 점검한다.
7. 결과는 캡슐과 승격 후보로 압축해 학습 자산으로 남긴다.

## 시작 레벨
### Level 0. Basic Conversation
- 대상: 빠른 질문, 짧은 설명, 단발 작업
- 읽을 것:
  - `START_HERE.md`
  - `POLICY.md`
- 특징:
  - 추가 모듈 없이도 시작 가능
  - 가장 가벼운 스타터 사용 방식

### Level 1. Guided Session
- 대상: 세션 흐름이 흔들리기 쉬운 사용자
- 추가 읽기:
  - `01_Modules/Madang_Pan_Lite_Starter/README.md`
- 특징:
  - 요청 분류
  - 짧은 진입 브리핑
  - 단일 현재 단계 유지

### Level 2. Semi Auto Session
- 대상: 반복 지시를 줄이고 싶은 사용자
- 추가 읽기:
  - `TASKS/SEMI_AUTO_PRESET_v0.1.md`
  - `TASKS/TEST_CASES_SEMI_AUTO_v0.1.md`
- 특징:
  - 시스템이 다음 단계를 스스로 이어감
  - 파일 변경이나 위험 단계에서는 멈춤
  - 완전 자동이 아니라 반자동 운영 패턴

### Level 3. Project Workspace
- 대상: 단일 세션을 넘는 작업
- 추가 읽기:
  - `01_Modules/Project_Workspace_Lite_Starter/README.md`
  - `TASKS/PROJECTS/README.md`
- 특징:
  - 별도 프로젝트 오케스트레이션 세션에서 사용하는 것을 권장
  - `BOOT_ENTRY`
  - `MISSION`
  - `ROADMAP`
  - `ORCHESTRATOR`
  - 4문서로 프로젝트 시작

### Level 4. Learning Loop
- 대상: 세션 결과를 자산으로 축적하고 싶은 사용자
- 추가 읽기:
  - `01_Modules/Learning_Loop_Lite_Starter/README.md`
- 특징:
  - 캡슐 기록
  - 승격 후보 기록
  - 자동 승격 없음

## 작업군 기준 운영
### BASIC
- 추천 흐름: 질문 수신 -> 답변
- 추가 모듈: 없음 또는 Madang Pan

### RESEARCH
- 추천 흐름: 질문 수신 -> IVK 회수 -> 근거 정리 -> 출력
- 추가 모듈: Madang Pan, Learning Loop

### WRITING
- 추천 흐름: 요구 정리 -> 초안 작성 -> 간단 리뷰 -> 출력
- 추가 모듈: Persona, Semi Auto, Learning Loop

### CODING
- 추천 흐름: 파일 탐색 -> 설계/수정 -> 검증 -> 결과 보고
- 추가 모듈: Coding Sequential, Security Gate, Semi Auto

### PROJECT
- 추천 흐름: 프로젝트 작업장 생성 -> MISSION -> ROADMAP -> ORCHESTRATOR -> 트랙 확장
- 추가 모듈: Project Workspace, Learning Loop, Security Gate

## 선택 모듈을 붙이는 방법
### Madang Pan Lite
- 언제 쓰나:
  - 세션 흐름이 자주 흐트러질 때
  - 초보 사용자에게 입구를 단순화하고 싶을 때
- 무엇을 더하나:
  - 분류
  - 짧은 브리핑
  - 단일 단계 유지

### Semi Auto Preset
- 언제 쓰나:
  - 반복적인 다음 단계 지시를 줄이고 싶을 때
  - 조사, 초안, 리뷰, 코딩 흐름을 반자동으로 잇고 싶을 때
- 무엇을 더하나:
  - 자동 전진
  - 멈춤 조건
  - 평가/학습 루프 연결

### Security Gate
- 언제 쓰나:
  - 외부 자료를 자주 받아들일 때
  - 코드/명령/외부 반출이 포함될 때
- 무엇을 더하나:
  - Inbound 점검
  - Execution 점검
  - Outbound 점검

### Project Workspace Lite
- 언제 쓰나:
  - 세션을 넘는 작업이 생길 때
  - 프로젝트 관리 문서가 필요할 때
- 무엇을 더하나:
  - 프로젝트별 정식 작업장
  - 4문서 시작 구조

### Learning Loop Lite
- 언제 쓰나:
  - 세션 교훈을 자산으로 남기고 싶을 때
  - 스킬/에이전트 승격 후보를 분리하고 싶을 때
- 무엇을 더하나:
  - 캡슐 작성
  - 승격 후보 작성
  - 세션 종료 루틴

## IVK의 위치
- IVK는 메모리 시스템이 아니라 로컬 회수 엔진으로 본다.
- 기본 역할:
  - 과거 작업 회수
  - 사용자 선호 회수
  - 유사 실패/성공 패턴 회수
  - 보안 경고 패턴 회수
- 원칙:
  - 먼저 로컬
  - 부족하면 외부 검색
  - 외부 검색은 자동 병합하지 않음

## 외부 자산 자비스화 원칙
- 공개된 외부 자산은 그대로 복제해 넣기보다, 현재 스타터의 규칙과 작업 방식에 맞게 재구성하는 것을 권장한다.
- 목표는 `수입(import)`이 아니라 `자비스화(Jarvis-ization)`다.

### 기본 규칙
1. 외부 자산을 그대로 코어에 복사하지 않는다.
2. 먼저 그 자산의 개념, 구조, 역할, 체크리스트를 분리해 본다.
3. 현재 스타터의 레이어 구조에 맞는지 판단한다.
4. 필요하면 일부만 흡수하고, 필요하면 완전히 새 문서로 다시 쓴다.
5. 반복 검증 전에는 `TASKS`나 실험 모듈에 두고, 나중에만 `AGENTS`나 `SKILLS`로 승격한다.

### 흡수 방식
- `개념만 가져오기`
  - 외부 자산의 핵심 아이디어만 추출
- `구조만 가져오기`
  - 역할 분리, 단계 흐름, 템플릿 구조만 참고
- `선택적 재작성`
  - 필요한 부분만 현재 스타터 문맥에 맞게 다시 작성
- `신규 자산화`
  - 외부 자산을 계기로 완전히 새로운 자비스용 자산 생성

### 금지 방향
- 외부 자산을 문맥 없이 그대로 붙여넣기
- 외부 시스템의 런타임 가정을 그대로 강제하기
- 현재 스타터의 코어 경계를 무시한 직접 이식
- 검증되지 않은 외부 운영철학을 즉시 기본값으로 채택하기

## 보안의 위치
- 보안은 부가 설명이 아니라 실제 게이트여야 한다.
- 최소 위치:
  - 입력 직후
  - 실행 직전
  - 출력 직전
- 원칙:
  - 위험을 설명만 하지 말고, 필요 시 흐름을 멈춘다.

## 브라우저의 위치
- 브라우저는 기본 전제가 아니라 선택형 보조 수단이다.
- 기본 흐름은 로컬 문서와 IVK만으로도 유지되어야 한다.
- 웹 탐색이 필요할 때만 브라우저를 추가로 사용한다.
- 브라우저 자동화는 스타터 기본값이 아니라 환경 허용 시의 확장 능력으로 본다.

## 학습의 위치
- 학습은 모든 작업군에 붙는 후행 루프다.
- 기본 흐름:
  - 결과 요약
  - 실패/막힘 요약
  - 재사용 포인트 추출
  - 캡슐 기록
  - 승격 후보 기록
- 원칙:
  - 기록은 먼저
  - 승격은 나중
  - 자동 승격은 하지 않음

## 공개용 스타터 원칙
1. 설치 없이 문서만으로 시작 가능해야 한다.
2. 구현을 강제하지 않아야 한다.
3. 사용자의 개인 시스템을 대체하지 않아야 한다.
4. 복잡한 자동화보다 안전한 기본값을 우선한다.
5. 선택 모듈은 기본 흐름을 오염시키지 않는 방식으로만 붙인다.

## 권장 공개 진입 순서
1. `README.md`
2. `START_HERE.md`
3. `POLICY.md`
4. 필요한 경우에만 선택 모듈 README

## 예시 시나리오
### 예시 1. 가장 가벼운 사용
- 사용자는 스타터를 열고 질문한다.
- 에이전트는 코어 규칙만 읽고 답한다.
- 추가 자산은 만들지 않는다.

### 예시 2. 반자동 리서치
- 사용자는 세미 오토로 조사 요청을 한다.
- 에이전트는 IVK를 먼저 조회한다.
- 로컬 근거를 정리해 보고한다.
- 끝날 때 캡슐 후보만 남긴다.

### 예시 3. 프로젝트 시작
- 사용자는 메인 세션이 아니라 별도 오케스트레이션 세션을 연다.
- 새 세션에서 프로젝트 작업장을 연다.
- `BOOT_ENTRY`, `MISSION`, `ROADMAP`, `ORCHESTRATOR`를 채운다.
- 필요할 때만 보안, 학습, 세미 오토를 얹는다.

## 최소 실패 방지 규칙
1. 코어 문서를 작업 산출물로 오염시키지 않는다.
2. 선택 모듈을 기본 의무처럼 강제하지 않는다.
3. 긴 로그보다 짧은 캡슐을 남긴다.
4. 자동처럼 보이더라도 위험 단계에서는 반드시 멈춘다.
5. 스타터를 제품처럼 고정하지 않고, 사용자가 자기 방식으로 가져갈 수 있게 둔다.

## 차기 승격 기준
- 여러 프로젝트에서 반복 사용 가능
- 공개 사용자에게 진입 장벽을 높이지 않음
- 특정 런타임을 강제하지 않음
- 설명 문서만으로도 오해 없이 사용 가능
